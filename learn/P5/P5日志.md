# T5.6：进程生命周期管理
## Chrome 无法打开窗口:
日志里的关键错误：

```
Authorization required, but no authorization protocol specified
Could not open the default X display
```

原因：`sudo python3 tls_capture.py --auto` 以 root 身份 spawn Chrome，但 root 没有当前用户的 X11 显示权限。Frida 的 `device.spawn()` 继承了 root 的环境，缺少 `DISPLAY` 和 `XAUTHORITY` 变量。**`device.spawn()` 以 root 身份启动 Chrome，但 X11 显示授权属于用户 palm**。传环境变量解决不了，因为进程 UID 是 root，X server 拒绝 root 连接。
初步解决方案:不要用 `device.spawn()` 启动 Chrome。改用 `subprocess.Popen` 以原始用户身份启动，然后 Frida attach。然后用现有的 `find_chrome_network_pid()` 等待 NetworkService 出现后 `device.attach()`。这样 Chrome 以 palm 身份运行（有 X11 权限），Frida 以 root 身份 attach（有 ptrace 权限），两不冲突。

代价是失去了 spawn 模式的"零时序差"优势，但 P4 已经证明 attach + 条件等待 correlator 的方案可以达到很高的命中率。Chrome 启动到 NetworkService 就绪通常有 3-5 秒，fd_tracker 已经先启动好了，时序窗口很小。

# 一、fd 精准匹配为什么失败

---

## 1. 从运行日志看，fd 根本没有拿到

capture.log 中从头到尾打印的 key 行几乎都是这种模式：

# five_tuple=... pid=105616 fd=-1

[KEY # ... hkdf/prf ->...]

说明 _handle_key() 接收到的 payload.fd 就是 -1。

而 _handle_key() 里关联流程是这样的：

dst_ip, dst_port, _method = correlator.find_connect(

    fpid, fd, wait_timeout=args.connect_wait_timeout

)

只有当 fd > 0 时，correlator.find_connect() 才会先走 _find_by_fd_locked()；否则直接退回 _find_by_time_locked()。

也就是当前实际上执行的是：

- fd <= 0

- 跳过 fd 精准匹配

- 直接用时序匹配

- 同一个 client_random 后续几条 secret 再走 cache

所以 fd精确=0 是必然结果，不是偶发。

---

## 2. 当前代码里其实已经写明了这个限制

你当前 chrome_hooks.js 里有非常关键的一段注释：

/**

 * 尝试读取 fd（ssl_ptr → rbio → fd）

 * 注意：HKDF 的 args[0] 不是 ssl_st*，此函数在 HKDF 上下文中通常返回 -1

 */

function readFd(ssl) {

    try {

        const rbio = ssl.add(RBIO_OFFSET).readPointer();

        if (rbio.isNull()) return -1;

        const fd = rbio.add(BIO_NUM_OFFSET).readS32();

        return (fd >= 3 && fd <= 65535) ? fd : -1;

    } catch (_) {

        return -1;

    }

}

这个注释本身几乎已经把根因说出来了：

> HKDF hook 里的 args[0] 并不是真正的 ssl_st*，所以用 ssl + rbio_offset + bio_num_offset 这条链去取 fd，本来就大概率失败。

而你本次测试里绝大多数 key 都来自 hkdf：

- hkdf = 392

- prf = 27

所以只要 HKDF 路径拿不到 fd，整体上就很难出现 fd 精准匹配。

---

## 3. 为什么连 PRF 也没有出现 fd 命中？

这点比 HKDF 更值得注意，因为 PRF 路径理论上 args[0] 更像 ssl_ptr。

但实际统计依旧是 fd精确=0，说明：

### 至少本次测试中，PRF 路径也没有提供有效 fd

可能原因有几类，但从“暂不改代码，只分析”的角度，我认为最可能的是下面两种：

---

### 原因 A：PRF 路径的 args[0] 虽然像 ssl_ptr，但当前 ssl_st_rbio / bio_st_num 偏移链不适用于这里

你的配置文件里写的是：

- ssl_st_rbio = 0x240

- bio_st_num = 0x03c

但这个链在当前 PRF hook 场景下未必成立，或者：

- ssl_ptr 不是你以为的那个结构层

- 当前 BoringSSL 在这个调用点结构不同

- rbio 不在这条链上

- 或读出来的是非法值，被过滤成 -1

---

### 原因 B：PRF 数量太少，而且日志输出没有单独验证 fd 是否曾经成功过

本次 prf 只有 27 条，数量很少。

但更关键的是，日志里没有出现过任何：

- fd=xx

- 或 fd精确 > 0

所以可以基本排除“只是 PRF 样本少”的说法，应该是PRF 上也没读到有效 fd。

---

## 4. eBPF 和五元组关联本身其实是正常的

这一点非常重要：你不要把“fd 精准匹配失败”误判成“五元组关联失败”。

因为从统计看：

- 密钥: 419 条

- 五元组命中: 419/419

- 关联方式: fd精确=0 时序=106 缓存=313

- connect 事件: 375

- 唯一连接: 106

这说明当前系统行为是：

### 第一步：第一次遇到某个 client_random

- 走时序匹配

- 命中一个 connect 事件

- 建立 cr_to_tuple 缓存

### 第二步：同一个 client_random 的后续 secret

- 直接从缓存取 tuple

- 不需要再次关联

所以：

- 五元组是成功的

- 只是成功方式不是 fd 精准匹配，而是时序 + cache

---

## 5. 因此，fd 精准匹配失败的根因总结

我把这部分压缩成一句话：

> 当前 hook 点上下文里拿不到真正可用的 ssl_st* → rbio → fd 路径，导致 hook 发回来的 fd 恒为 -1，于是关联器完全无法触发 _find_by_fd_locked()，只能依赖时序匹配和缓存。

换句话说：

- fd_tracker 没问题

- correlator 没问题

- 主要问题在 hook 侧传不出有效 fd

# 二、为什么会“漏捕”部分密钥

这个问题其实已经可以非常精确地下结论了。

---

## 1. 实际缺失的是哪些？

我做了集合对比，结果是：

- missing_count = 79

- extra_count = 0

- missing_by_label = {'CLIENT_RANDOM': 79}

也就是：

> hook 没有多抓错抓；只是 env 有 79 条 CLIENT_RANDOM，hook 没抓到。

这一点很关键，因为它排除了很多可能性：

### 被排除的情况

- 不是 HKDF 参数错了

- 不是 TLS 1.3 标签漏了

- 不是 exporter 漏了

- 不是 secret 长度读取错了

- 不是 hook 输出有脏数据

- 不是 hook 比 env 多了假阳性

---

## 2. 这些缺失的 79 条，全部是“只有一条 CLIENT_RANDOM”

我统计了缺失的 client_random 分组，发现：

- missing_unique_cr = 79

- 每个组长度都是 1

也就是每一个缺失样本，都是：

CLIENT_RANDOM <cr> <master_secret>

只有这一条，没有同一 cr 的 TLS 1.3 五件套。

这说明这些缺失条目本质上属于：

> TLS 1.2 / PRF 路径的会话

而不是 TLS 1.3 会话的一部分。

---

## 3. 为什么这些 TLS 1.2 会话 env 有、hook 没有？

这部分我认为要区分“技术原因”和“现象特征”。

---

## 4. 现象特征：这些缺失的 master secret 大量重复

你给的 env log 里能看到很多类似模式：

CLIENT_RANDOM xxxx 21aef74707ed7141e264...

CLIENT_RANDOM yyyy 21aef74707ed7141e264...

CLIENT_RANDOM zzzz 21aef74707ed7141e264...

也就是说：

- 很多不同的 client_random

- 对应完全相同的 master_secret

这非常不正常，如果是普通完整 TLS 1.2 全握手，会话之间 master secret 不应如此大规模重复。

这说明这些缺失项更像是：

> 恢复/复用类会话

> - Session resumption

> - Ticket/PSK 复用

> - 某种恢复路径下重复使用既有 secret

> - 或 keylog 层补记了复用会话，而你的 PRF hook 并没有再次经过真正的 PRF 派生

这是一个非常强的信号。

---

## 5. 这意味着什么？

意味着环境变量 SSLKEYLOGFILE 记录的是：

> 所有浏览器认为对解密有用的最终 keylog 条目

而你当前 hook 的 PRF 路径记录的是：

> 只有真正走到了你挂钩的 PRF 计算函数的 TLS 1.2 会话

所以两者天然不完全等价。

---

## 6. 换句话说：你现在缺的不是“普通 PRF 全握手”，而更像是“恢复型 TLS 1.2 会话”

因为如果是完整 TLS 1.2 握手漏掉 79 次，那么通常会表现为：

- 某些连接完全没抓到

- 同时有新的 master_secret

- 不会大批量出现“不同 client_random 对应同一个 master_secret”的情形

而你这里恰恰看到了大量重复 master secret，这更像：

### 环境变量层能记住恢复会话

浏览器内部 keylog 逻辑知道：

- 这个连接虽然没有重新跑完整 PRF

- 但它仍然对应一个 CLIENT_RANDOM + 既有 master_secret

- 所以会把这条也写出去

### 但你的 hook 只挂在 PRF / HKDF 派生点

如果这个恢复会话没有重新走到 PRF 派生函数，那么：

- env 会有

- hook 不会有

这和你之前在 P3/P4 文档里已经碰到的结论高度一致：

恢复/票据/PSK 类连接经常是 hook 捕获的盲区，而 keylog 输出点能覆盖它们。

---

## 7. 为什么这次没有像以前那样表现成“缺一个 client_random 对应 5 条”？

因为这次缺失的是 TLS 1.2，会话只有一条：

CLIENT_RANDOM <cr> <master_secret>

不像 TLS 1.3 那样同一个会话有 5 条左右：

- CHS

- SHS

- CTS0

- STS0

- EXPORTER

所以这里 79 条缺失 = 79 个 TLS 1.2 缺失会话，并不是 79/5 这样的换算。

---

## 8. 为什么 extra_count = 0 很重要？

这说明：

> hook 当前抓到的内容都是真实正确的，没有产生 env 不存在的伪条目。

所以你当前问题不是“抓错”，而是“覆盖面不够”。

换句话说：

- 现有 HKDF / PRF 实现的准确率是高的

- 但是对 TLS 1.2 恢复类场景 的召回率不够

# 三、把两个问题合在一起看，会得到什么更深的结论

这两个现象其实是相关的：

---

## 1. fd 精准匹配失败

说明你当前 Hook 点虽然能抓 key，但不适合作为稳定的连接语义入口。

尤其 HKDF hook：

- 更偏“底层密钥派生”

- 不一定拿得到连接对象

- 不一定能拿到 fd

- 更适合抓 secret，不适合做精确连接归属

---

## 2. env 比 hook 多出 79 条 CLIENT_RANDOM

说明你当前 Hook 点虽然能覆盖大量正常派生，但不能覆盖所有“浏览器最终会写出的 keylog 事件”。

尤其是：

- TLS 1.2 恢复/复用类

- 或某些不再重新进入 PRF 的路径

---

## 3. 这两个现象共同指向一个方向

我先只分析，不提修改方案，单讲方向判断：

> 你当前的 Hook 选点（PRF/HKDF）更像“密钥派生观察点”，而不是“最终秘密输出点”。

也就是说，它适合：

- 提取大多数 secret

- 做版本特征分析

- 做 KDF 指纹数据库

但它未必天然适合：

- 100% 覆盖浏览器最终 keylog 输出

- 稳定拿 fd / 连接对象

- 精准做到所有恢复路径

这个判断其实和你长期想结合 BoringSecretHunter 的方向是吻合的。
