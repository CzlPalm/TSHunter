# 阶段 0：源码阅读与整合策略分析

## 0.1 TLSKeyHunter 核心理解

### 1. HKDF 识别流程

TLSKeyHunter 的 HKDF 识别核心目标字符串是 `"s hs traffic"`。

主流程大致如下：
1. 先在 Ghidra 已识别的字符串数据里直接查找完全匹配的 `"s hs traffic"`
2. 如果字符串存在且有 XREF，则取引用该字符串的地址，再定位其所在函数
3. 如果字符串存在但没有直接引用，则尝试在 `.data.rel.ro` / `.rdata` 里寻找指向该字符串的指针，再继续追踪引用该指针的函数
4. 如果普通字符串搜索失败，则进入 `.rodata` / `.rdata` 的字节模式扫描回退：
   - 将目标字符串转成字节
   - 同时搜索大端/小端形式，以及追加 `0x00` 或空格的变体
   - 找到命中地址后，对该地址做 XREF 分析
5. 若 XREF 指向的不是函数内部，而是另一个数据段，则继续向后/向前追踪数据指针，直到找到函数引用
6. 此外还实现了对“栈字符串”的补充检测，即扫描指令中是否把形如 `"art sh s"` 这类倒序常量搬到栈上，用于覆盖某些编译器把 label 内联到栈上的情况

结论：HKDF 路径是 TLSKeyHunter 中最完整、最健壮的一条识别链路，包含：
- 普通字符串搜索
- `.data.rel.ro` 指针补追
- `.rodata` 字节模式回退
- 栈字符串回退

### 2. PRF 识别流程

PRF 的核心目标字符串是 `"master secret"`。

其主流程与 HKDF 类似，但不完整：
1. 先在 Ghidra 定义的字符串里查找 `"master secret"`
2. 如果命中字符串且存在 XREF，则追踪到引用点，再分析该字符串附近的调用关系
3. 也支持 substring 情况的辅助逻辑：当 `"master secret"` 是更大字符串的一部分时，可尝试基于偏移地址找引用
4. 也支持栈字符串搜索，即寻找类似 `"s retsam"` 的立即数装载

**关键缺陷**：
- PRF 没有像 HKDF 那样成熟的 `.rodata` 字节模式回退主路径
- 因而在 Chrome/BoringSSL 这种“普通字符串引用不明显，但 `.rodata` 中字节存在”的场景下容易失败
- 同时，`"master secret"` 作为 `"extended master secret"` 的子串，会引入错误命中风险

这与说明文档中的失败诊断完全一致。

### 3. 指纹提取逻辑

TLSKeyHunter 的指纹提取不是固定长度，而是“从函数入口开始读取，直到遇到合适的停止条件”。

主要逻辑：
1. 从目标函数入口开始遍历指令
2. 使用 `getLengthUntilBranch(...)` 计算应该提取多少字节
3. 停止条件主要是遇到：
   - `jump`
   - 条件跳转
   - `call`
4. 但为了避免过早截断，代码还做了多种特殊处理：
   - 若调用目标仍在当前函数内部且距离很近，则继续，不把它当停止点
   - 某些 `call $+...` 形式跳过
   - 如果函数太短，怀疑只是 wrapper，则尝试跟随 `jmp`/GOT 跳转到真实实现
   - 对 Go、某些特殊分支、跨 span 场景有若干额外分支码（如 `-42/-43/-44/-66`）做递归再分析
5. 最终通过 `readBytes(memory, entryPoint, numBytes)` 从函数入口连续读出字节，并格式化为十六进制空格串

因此它提取的是“函数起始稳定前缀字节”，而不是整个函数。

### 4. RVA 计算逻辑

TLSKeyHunter 输出两类地址：
- Ghidra 地址：直接显示函数入口地址
- IDA base 0x0 偏移：通过 `get_ida_address(...)` 计算

计算方式：
- 32 位默认减去 `0x00010000`
- 64 位默认减去 `0x00100000`

也就是把 Ghidra 默认 imageBase 转成 IDA 常用的 base 0 偏移。
这与指令文件里的约束一致：Chrome ELF 64 位通常按 `0x00100000` 处理，最终输出要用 IDA 风格 RVA。

### 5. 输出格式

TLSKeyHunter 当前不是结构化输出，而是面向终端的人类可读日志，核心包括：
- 识别到的函数名/标签
- 函数签名
- `Function offset (Ghidra)`
- `Function offset (IDA with base 0x0)`
- `Byte pattern for frida`
- 某些情况下还会打印 wrapper/间接调用/参数索引等辅助信息

所以它已经具备我们需要的三个核心数据：
- function
- rva
- fingerprint

但还没有统一的 `[RESULT] ...` 机器可解析格式。

## 0.2 BoringSecretHunter 核心理解

### 主脚本文件

`BoringSecretHunter` 目录下存在多个 `.java` 文件，其中核心 Ghidra 脚本是：
- `TLShunter/BoringSecretHunter/BoringSecretHunter.java`

另外 `src/.../ghidra_scripts/` 下也有同名脚本副本/打包版本，但根目录这个文件已经足以代表当前主逻辑。

### 1. 如何定位 `ssl_log_secret`

BoringSecretHunter 主要通过与 TLS key log 相关的字符串反推 `ssl_log_secret()`。

主流程如下：
1. 首先搜索主字符串 `"EXPORTER_SECRET"`
2. 如果没找到，则回退搜索 `"CLIENT_RANDOM"`
3. 先在 Ghidra 已定义字符串里搜索
4. 若失败，则在只读数据段中做十六进制字节模式搜索
   - 支持 `.rodata` / `.rdata` / `__cstring` / `__const`
   - 比 TLSKeyHunter 更通用，兼容 ELF/PE/MachO
5. 对命中的字符串地址做 XREF 分析，拿到引用该字符串的函数
6. 然后从“使用该字符串的函数”继续寻找真正被调用的函数：
   - 在字符串引用点之后向前/向后找 `call`
   - 取该 `call` 的目标作为候选目标函数
7. 如果无法直接解析调用目标，则退而求其次：
   - 根据 XREF 数量判断是否直接使用当前函数
   - 或者找它的 caller，把 caller 当作 `ssl_log_secret`
8. 对 Rust/Rustls 还包含额外分支，但对本项目核心目标 BoringSSL 来说，主路径就是 `EXPORTER_SECRET` / `CLIENT_RANDOM` → 字符串引用函数 → 调用目标函数

结论：BoringSecretHunter 对 `ssl_log_secret` 的定位更偏“字符串 → 调用链”分析，而非只停留在“字符串被哪个函数引用”。

### 2. 指纹提取逻辑

BoringSecretHunter 的思路与 TLSKeyHunter **相似，但更简单**：
1. 确定目标函数入口
2. 从入口开始遍历指令
3. `getLengthUntilBranch(function)` 在遇到以下流控制时停止：
   - jump
   - conditional
   - call
4. 若整个函数没有分支，则返回特殊值，再尝试用 caller 替代
5. 读取从函数入口到停止点之间的连续字节
6. 格式化为十六进制模式串输出

与 TLSKeyHunter 对比：
- 相同点：都是“函数前缀字节指纹”，都按“直到第一个合适分支/调用”为止
- 不同点：TLSKeyHunter 增加了更多 wrapper / span / indirect call / 多架构细节处理，明显更复杂、更健壮

### 3. 输出格式

BoringSecretHunter 同样以终端日志输出，核心字段包括：
- `Function label`
- `Function offset (Ghidra)`
- `Function offset (IDA with base 0x0)`
- `Byte pattern for frida (friTap)`

它还把部分结果缓存在静态变量里，如：
- `tls13GhidraOffset`
- `tls13IdaOffset`
- `tls13BytePattern`

但总体上仍然不是统一 JSON 或 `[RESULT]` 机器格式。

### 4. Docker 配置与 TLSKeyHunter 的异同

**共同点：**
- 都基于 `gradle:jdk21`
- 都安装 Python、file、构建工具
- 都使用 Ghidra headless 执行 `MinimalAnalysisOption.java` + 主脚本
- 都输出到容器日志，且挂载 `/host_output`

**BoringSecretHunter 更完善的地方：**
- Ghidra 版本更高：`12.0.3`，而 TLSKeyHunter 用的是 `11.1.2`
- 只读数据段兼容性更强，覆盖 `.rodata/.rdata/__cstring/__const`
- `ghidra_analysis.sh` 支持：
  - ELF / Mach-O / PE32 / raw data
  - APK / IPA / ZIP 预解包
  - 原始数据文件自动选择 processor
  - 大文件模式选择（normal/fast/skip）
  - 环境变量控制调试与大文件模式
- Dockerfile 会复制整个项目并做 `pip install .`

**TLSKeyHunter 更简单的地方：**
- 容器更轻
- 脚本更直接聚焦 HKDF/PRF
- 但平台覆盖和预处理能力弱于 BoringSecretHunter

## 0.3 对比分析与整合策略

### 1. 两个项目有哪些共享逻辑？

共享逻辑很多，主要集中在以下几类：

#### a. 字符串搜索
两者都实现了：
- 遍历 Ghidra 已定义字符串
- 查找目标字符串
- 找不到时转向只读数据段字节模式搜索

#### b. XREF 追踪
两者都实现了：
- 根据字符串地址取 references
- 根据引用点找到所在函数
- 如果引用点不在函数内，则尝试继续追踪数据指针/间接引用

#### c. 指纹提取
两者都实现了：
- 从函数入口读字节
- 用“遇到分支/调用则停”的策略截取前缀
- 输出为十六进制空格串

#### d. RVA 计算
两者都使用同一套思路：
- 32 位减 `0x10000`
- 64 位减 `0x100000`
- 输出 IDA base 0 风格偏移

#### e. Headless Ghidra 运行方式
两者都使用：
- `MinimalAnalysisOption.java`
- `analyzeHeadless`
- prescript + postScript 的组合

也就是说，这两个项目并不是完全不同的实现，而是高度同构的两个脚本分支。

### 2. 哪个项目更适合作为基础项目被扩展？

**结论：更适合以 `TLSKeyHunter` 作为脚本基础，以 `BoringSecretHunter` 作为运行环境/Docker 基础。**

如果必须只选一个“主基础项目”，我推荐：

**代码分析逻辑基础：`TLSKeyHunter`**

原因：
1. **代码组织虽然更杂，但分析能力更强**
   - 尤其是 HKDF/PRF 场景下，wrapper、间接调用、栈字符串、span 分析等逻辑都已经在里面
2. **字符串搜索 + XREF 追踪更贴近本项目目标**
   - 目标之一就是修 PRF，而 PRF 现成逻辑只在 TLSKeyHunter 里
3. **更容易扩展新的 KDF 类目标**
   - HKDF、PRF、key_expansion 本质上是一类问题：都是 label → 引用 → 候选函数 → 指纹
4. **已有 `.rodata` 回退样板可直接迁移到 PRF/key_expansion**
   - HKDF 那条成功路径正是阶段 2 需要复制到 PRF 的基础

但同时：

**运行环境/工程封装基础：`BoringSecretHunter`**

原因：
1. Docker/Ghidra 版本更新
2. 格式兼容范围更广
3. 分析脚本入口更完善
4. 对多格式输入、raw dump、大文件处理更成熟

因此最合理的不是“整包基于某一个项目”，而是**二层整合**：
- **脚本分析核心**：以 TLSKeyHunter 为主
- **容器与执行框架**：借鉴 BoringSecretHunter

### 3. 推荐的整合方案

推荐方案如下：

#### 方案：以 TLSKeyHunter 的分析框架为核心，合并 BoringSecretHunter 的 `ssl_log_secret` 识别逻辑，并采用 BoringSecretHunter 的 Docker/运行器设计思路。

具体理由：

1. **当前整合目标中，最复杂、最关键的是 HKDF/PRF/key_expansion**
   - 这些都天然属于 TLSKeyHunter 的问题域
   - 尤其阶段 2 的 PRF 修复，必须建立在 TLSKeyHunter 已有逻辑之上

2. **`ssl_log_secret` 是一个相对独立的字符串追踪任务**
   - BoringSecretHunter 的能力可以较容易并入 TLSKeyHunter
   - 实现上只需要增加一个新的目标字符串集合和一段调用链解析逻辑

3. **TLSKeyHunter 已经有更丰富的函数候选修正机制**
   - wrapper 识别
   - GOT / 间接跳转
   - stack string
   - 多阶段 fallback
   这些对 Chrome/BoringSSL 更关键

4. **BoringSecretHunter 的优势更多体现在工程封装而不是核心识别算法**
   - 因而最优做法不是把 KDF 逻辑塞进 BoringSecretHunter，而是把它的 Docker/脚本外壳迁入新项目

5. **共享逻辑应抽成统一基础设施**
   整合后的 `TLShunterAnalyzer.java` 里应抽出：
   - 只读数据段枚举
   - 字符串/字节模式搜索
   - XREF → 函数候选收集
   - wrapper 处理
   - 指纹提取
   - IDA 风格 RVA 计算

6. **执行顺序建议与指令文件一致**
   - HKDF
   - ssl_log_secret
   - PRF
   - key_expansion

这样可以保证：
- 先保住 TLS 1.3 的成熟路径
- 再引入 `ssl_log_secret`
- 再在同一套基础设施里修 PRF 和新增 key_expansion

## 阶段 0 最终结论

### 结论摘要

- `TLSKeyHunter` 更适合作为**分析脚本逻辑基础**，因为它已经覆盖 HKDF/PRF 核心问题，且具备更复杂的候选函数修正逻辑
- `BoringSecretHunter` 更适合作为**工程执行环境参考**，因为它的 Docker、Ghidra 版本、多格式输入支持更完善
- 最佳整合路线不是简单二选一，而是：
  - **以 TLSKeyHunter 的分析逻辑为核心**
  - **吸收 BoringSecretHunter 的 ssl_log_secret 识别链路**
  - **借鉴 BoringSecretHunter 的 Docker/运行器设计**

### 当前实际落地架构补充说明

你问的这个问题，当前答案是：**不是通过代码直接调用原始 `TLSKeyHunter.java` 或 `BoringSecretHunter.java`**，而是采用了“**重新实现一个统一脚本**”的方式整合两者。

也就是说，当前架构不是：
- 在新项目里 import / 调用原始 `TLSKeyHunter` 类
- 或在 Java 里串行执行两个已有脚本

而是：
- 新建一个独立的整合脚本 `integrated/scripts/TLShunterAnalyzer.java`
- 把两个项目里真正需要的识别思路，抽象成新的共享基础设施和新的分析入口
- 再由这个统一脚本一次性完成 HKDF、`ssl_log_secret`、PRF、`key expansion` 的识别

### 当前整合方式分层

#### 1. 脚本层：重写式整合，不直接调用原项目主脚本

当前 `TLShunterAnalyzer.java` 是一个新的 Ghidra headless 脚本，它：
- 自己维护统一的结果结构 `ResultRecord`
- 自己维护统一的输出格式 `[RESULT] ...`
- 自己维护共享方法，例如：
  - 字符串搜索
  - 只读段扫描
  - XREF → 函数收集
  - 调用点后续函数解析
  - 指纹提取
  - RVA 计算

因此它对原项目的“复用”主要体现在**逻辑借鉴**，不是“类级别直接调用”。

#### 2. HKDF / PRF / key_expansion 这一支，主要继承 TLSKeyHunter 的思路

当前统一脚本里，这一支的设计来源于 TLSKeyHunter：
- 先找字符串
- 再找 XREF
- 必要时扫只读段
- 找到函数后提取前缀指纹
- 输出 RVA + fingerprint

其中：
- `HKDF` 继续沿用 KDF 类目标的通用入口
- `PRF` 在阶段 2 里已经升级为专用识别逻辑
- `KEY_EXPANSION` 也走和 PRF 相近的 label → XREF → 函数 识别链

#### 3. `ssl_log_secret` 这一支，主要吸收 BoringSecretHunter 的思路

`ssl_log_secret` 没有直接调用原始 `BoringSecretHunter.java`，而是把它的关键方法论迁移到了新脚本中：
- 搜 `EXPORTER_SECRET`
- 搜不到时回退 `CLIENT_RANDOM`
- 先拿“引用该字符串的函数”
- 再在该引用点后面寻找被调用函数
- 把被调用函数作为 `ssl_log_secret` 候选

因此这里复用的是**调用链识别策略**，不是复用旧类本身。

#### 4. 运行层：借鉴 BoringSecretHunter 的工程外壳

运行层也不是直接启动原始项目，而是重新搭了新的整合工程：
- `integrated/Dockerfile`
- `integrated/ghidra_analysis.sh`
- `integrated/run.py`

其中：
- `Dockerfile` 借鉴的是 BoringSecretHunter 的较新 Ghidra 版本和容器组织方式
- `ghidra_analysis.sh` 负责在容器里执行一次 `analyzeHeadless`
- `run.py` 负责：
  - 构建镜像
  - 启动容器
  - 解析 `[RESULT]` 行
  - 组装 JSON

### 当前实际调用链

当前完整调用链是：

1. 用户执行 `integrated/run.py`
2. `run.py` 构建并运行整合 Docker 镜像
3. 容器内执行 `integrated/ghidra_analysis.sh`
4. `ghidra_analysis.sh` 调用 Ghidra `analyzeHeadless`
5. Ghidra 运行：
   - `MinimalAnalysisOption.java`
   - `TLShunterAnalyzer.java`
6. `TLShunterAnalyzer.java` 顺序执行：
   - HKDF
   - `SSL_LOG_SECRET`
   - PRF
   - `KEY_EXPANSION`
7. 脚本输出统一的 `[RESULT]` 行
8. `run.py` 解析结果并写出 JSON

### 为什么当前不直接调用原始 `TLSKeyHunter.java`

主要有几个原因：

1. **原始脚本输出格式不统一**
   - 两个项目都是面向终端的人类可读输出
   - 不适合直接作为统一 JSON 生产管线的底层接口

2. **原始脚本内部结构不适合直接拼装**
   - 两边都有各自的全局状态、打印逻辑、分析流程控制
   - 直接互调会让状态管理和输出解析非常混乱

3. **需要共享基础设施，而不是简单串联**
   - 阶段 2 的 PRF 修复要求 `.rodata` 多命中、独立字符串判定、交叉验证
   - 这些更适合在一个统一脚本里集中实现

4. **整合目标是“一次分析、统一输出”**
   - 如果直接串行跑两个旧脚本，会更像“包装器”而不是“整合器”
   - 当前做法更符合后续扩展到统一 JSON 和批处理能力的目标

### 当前架构一句话总结

当前 TLShunter 的整合方式是：

**以 TLSKeyHunter 的 KDF 识别思路为主线，以 BoringSecretHunter 的 `ssl_log_secret` 调用链分析为补充，在新的 `TLShunterAnalyzer.java` 中重写整合，并通过新的 Docker + `run.py` 统一执行。**

### 对后续阶段的意义

这种架构的好处是：
- 后续修 PRF 不需要改原项目目录
- 可以持续在 `TLShunterAnalyzer.java` 里演进共享基础设施
- `run.py` 能稳定消费统一的 `[RESULT]` 输出
- 后续阶段 3/4/5 的 JSON 对齐、批处理、报告能力都更容易追加
