# P7-0：项目背景、目标、架构定位与长期规划

## 1. 文档目的

本文档用于向新的 AI 协作者快速说明项目背景、当前目标、两仓库分工、关键技术路线、阶段性约束与未来规划。

适用对象：
- Claude Code
- Cursor 内其他 AI
- 后续参与项目设计、实现、验证、批量分析、部署的协作者

本文档不聚焦某一轮具体 bug 修复，而是作为项目的“背景总览 + 长期规划说明”。

## 2. 项目总目标

本项目的最终目标是：

**基于 TLSKeyHunter 与 BoringSecretHunter 的技术路线，为不同浏览器/版本构建一个可持续维护的 TLS 密钥指纹数据库。**

数据库中的每条记录，理想上需要把以下信息与浏览器版本关联起来：
- 浏览器名称
- 精确版本号
- 平台
- 架构
- TLS 库类型
- 关键 hook 点的 RVA
- 对应函数指纹 / 特征码
- hook 点语义与参数说明
- 密钥提取策略
- 与网络连接对应的五元组关联策略
- 数据来源与验证状态

项目最终希望实现的能力是：

1. 给定一个浏览器二进制文件，能够识别其版本与环境；
2. 如果数据库中已有该版本记录，直接读取并输出可用 hook 信息；
3. 如果数据库中没有该版本记录，则尝试通过指纹 + 特征码 + RVA 漂移进行 relocate；
4. 如果 relocate 无法可靠得到完整 hook 点，则退回逆向分析；
5. 将分析结果写回数据库，形成可复用的版本知识；
6. 在真实运行时，依据数据库结果对目标浏览器进行 hook，提取 TLS 密钥；
7. 对提取到的密钥进一步做五元组关联，服务于实时观测与事后取证；
8. 在服务器上对大量浏览器版本做批量分析，持续扩充数据库。

## 3. 两个仓库的角色定位

当前项目分为两个主要仓库：
- `TSHunter`
- `p_t_c`

这两个仓库是上下游关系，而不是互相替代关系。

### 3.1 `TSHunter` 的定位

`TSHunter` 是“分析与知识生成侧”的核心仓库，主要负责：

1. 识别、分析浏览器二进制文件；
2. 判断数据库中是否已有该版本的 hook 记录；
3. 若数据库无记录，则优先尝试 relocate；
4. 若 relocate 结果不可靠，则进入完整逆向分析；
5. 产出标准化 JSON / hook 点结果；
6. 将结果入库，形成版本知识；
7. 为后续运行时 hook 仓库提供结构化数据来源。

简化理解：

**TSHunter = 浏览器 TLS hook 指纹知识的生产端 / 分析端 / 入库端。**

### 3.2 `p_t_c` 的定位

`p_t_c` 是“运行时消费与执行侧”的核心仓库，主要负责：

1. 识别当前需要 hook 的浏览器；
2. 从数据库中查询对应版本数据；
3. 若数据库无数据，则触发分析侧流程；
4. 若数据库有数据，则直接读取并用于运行时注入；
5. 按用户选择的模式进行 hook 与密钥提取；
6. 对结果进行五元组关联；
7. 支持单目标模式与批处理模式。

简化理解：

**p_t_c = 数据库消费端 / 运行时注入端 / 密钥抓取端 / 取证关联端。**

---

## 4. 两仓协作的理想闭环

理想闭环应当是：

1. `p_t_c` 识别到目标浏览器；
2. 查询数据库；
3. 若命中，直接使用对应 hook 数据进行注入；
4. 若未命中，则交由 `TSHunter` 流程处理；
5. `TSHunter` 优先尝试 relocate；
6. relocate 成功则自动入库；
7. relocate 不足则完整分析并入库；
8. `p_t_c` 再重新读取数据库结果并执行 hook；
9. hook 得到的 TLS 密钥与五元组完成关联；
10. 验证成功的版本被标记为高可信基线。

这个闭环是整个项目成型的关键。

---

## 5. 当前关键技术路线

项目当前的核心技术基础来自两个方向：

### 5.1 来自 TLSKeyHunter 的能力
- 基于 Ghidra 的静态逆向分析
- 对多种 TLS 实现进行 hook 点识别
- 生成函数指纹 / 特征码
- 提供分析脚本与基础识别能力

### 5.2 来自 BoringSecretHunter 的能力
- 面向 BoringSSL / Chrome 的关键点验证经验
- 与真实 TLS 密钥提取相关的语义信息
- 更贴近实际 hook 与密钥导出的经验数据
- 与 SSLKEYLOGFILE / Wireshark 等验证链路相关的基线知识

### 5.3 当前项目的创新整合方向
本项目不是简单复用上述两个项目，而是做以下工程化整合：

1. 提炼 hook 点为版本知识；
2. 构建数据库模型；
3. 为小版本漂移设计 relocate 机制；
4. 建立自动分析、自动入库、自动查询的最小闭环；
5. 向运行时 hook 仓库输出可消费数据；
6. 将五元组关联纳入统一工作流；
7. 形成可维护、可持续更新的浏览器 TLS 密钥知识库。

## 6. 当前项目阶段判断

截至 P7 阶段，项目已经不再是单纯的“跑一个 Ghidra 脚本”，而是在逐步形成一套完整系统：

### 已有能力
- 针对 Chrome/BoringSSL 产出较稳定的 4 个核心 hook 点；
- 已构建 SQLite 数据库模型；
- 已完成基础 ingest/query/capture 工作流；
- 已完成 fingerprint relocate 第一版工程化接入；
- 已能在 DB miss 时自动寻找 verified source 并尝试 relocate；
- relocate 不足时可回退到完整分析；
- 已具备批量分析雏形；
- 已开始形成阶段文档与架构文档。

### 当前仍未完全打通的部分
- 自动分析结果与手工基线 schema 仍不完全一致；
- hook 结果还缺少部分语义字段、参数说明与运行时知识；
- `p_t_c` 与 `TSHunter` 的自动联动尚未彻底打通；
- 批量下载、批量分析、批量校验、批量入库流程尚未完成；
- 五元组关联能力还未工程化落地成稳定工作流；
- 多浏览器、多 TLS 库、多平台扩展仍处于后续阶段。

## 7. 当前主要数据流

### 7.1 分析流
`二进制 -> 静态识别 -> hook 点提取 -> JSON -> 入库`

### 7.2 查询流
`浏览器识别 -> 数据库查询 -> 命中返回 hook 配置`

### 7.3 缺失流
`浏览器识别 -> 查询 miss -> relocate -> 不足则逆向分析 -> 入库 -> 回到查询`

### 7.4 运行时流
`读取 hook 配置 -> 注入浏览器 -> 提取 TLS 密钥 -> 五元组关联 -> 输出/存档`

---

## 8. 当前最重要的工程原则

### 8.1 保持“分析端”和“执行端”边界清晰
- `TSHunter` 不要过度承担运行时 UI 或交互逻辑；
- `p_t_c` 不要过度内嵌逆向分析逻辑；
- 两者通过数据库 / 标准 JSON schema 解耦。

### 8.2 数据结构优先于脚本堆叠
当前项目后续是否能规模化，关键不在于再写多少脚本，而在于：
- schema 是否统一；
- hook 结果是否标准化；
- 版本知识是否可复用；
- 运行时配置是否可自动生成。

### 8.3 验证与分析要分层
- 分析结果不等于验证完成；
- relocate 成功不等于真实抓取可靠；
- 需要保留 `verified`、`verified_method`、实测覆盖率等验证层数据。

### 8.4 优先跑通 Chrome/BoringSSL 主链路
当前应优先把：
- Chrome
- Linux
- x86_64
- BoringSSL
这一条主链路做深、做透、做稳定，再扩展到其他浏览器。

## 9. 用户当前意图的工程化表述

用户当前真正想做的，不只是“抓到某一次 TLS key”，而是：

**建立一个持续维护的浏览器 TLS 密钥指纹数据库系统。**

这个系统要能支持：
- 版本关联
- 指纹匹配
- relocate 漂移适配
- 分析回退
- 数据库存档
- 实时 hook
- 五元组关联
- 批量扩库
- 持续更新

因此，后续所有实现决策都应围绕“系统化、数据库化、持续维护”来设计，而不是停留在一次性脚本层面。

## 10. 推荐的整体路线图

### 阶段 A：稳定 Chrome/BoringSSL 主链路
- 修好 `TSHunter` 自动分析结果与手工基线之间的结构差距；
- 补齐 `p_t_c` 当前消费数据库的能力；
- 让两仓在单版本 Chrome 上形成完整闭环。

### 阶段 B：整合成统一工作目录
- 将 `TSHunter` 与 `p_t_c` 放入统一上层目录；
- 设计共享配置、共享数据库、共享 schema、共享脚本；
- 明确“分析服务”和“运行时消费服务”的边界。

### 阶段 C：批量分析 Chrome 版本
- 批量下载 Chrome 各版本二进制；
- 在服务器上批量分析；
- 形成第一批版本数据库；
- 检查跨小版本 relocate 效果。

### 阶段 D：运行时抓取与五元组关联落地
- 在 `p_t_c` 中完成 hook 执行主链路；
- 完成与数据库联动；
- 输出 TLS 密钥并完成五元组关联。

### 阶段 E：扩展到其他浏览器与 TLS 库
- Chromium 系派生浏览器
- 其他浏览器
- 不同 TLS 库
- 更多平台 / 架构

## 11. 风险与难点

### 11.1 relocate 不是万能的
对于某些版本差异较大的二进制，小版本也可能出现：
- 非统一漂移
- 多个 hook delta 不一致
- relocate 只能给出 `PARTIAL`
- 必须退回完整分析

### 11.2 静态分析结果不天然等于可抓取
需要区分：
- 找到函数位置
- 能否正确读取参数
- 能否获取目标密钥
- 能否与流量对齐验证

### 11.3 数据库 schema 一旦混乱，后续很难维护
必须尽快统一：
- 自动分析 JSON
- 手工基线 JSON
- 数据库表字段
- 运行时消费格式

### 11.4 五元组关联比 hook 本身更容易工程失真
需要明确：
- 依赖什么来源获取连接信息；
- 哪些字段实时采集；
- 哪些字段通过时间相关性推断；
- 如何对结果可信度做标记。

## 12. 给后续 AI 的一句话总结

如果你是新接手该项目的 AI，请把这个项目理解为：

**一个围绕浏览器 TLS key hook 点展开的“版本知识库 + 自动分析 + 运行时消费 + 五元组取证”的长期工程，而不是一次性逆向脚本集合。**

请优先保证：
- schema 清晰
- 主链路闭环
- 两仓职责明确
- 数据可复用
- 批量扩展可持续
