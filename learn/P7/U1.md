# U1：S1 审计、S2 schema 设计与真正 U1 包结构整理结果

## 1. 文档目的

本文档记录以下三个阶段在当前仓库中的实际执行结果：

1. **S1**：审计 `tls_capture` 运行时到底消费哪些字段；
2. **S2**：设计统一 schema 与 `profile` 边界；
3. **U1**：真正开始整理包结构、入口、`import`、`pyproject`、CLI；
4. 给出 **U2** 的明确进入条件与后续收口路径。

本文档是对 `U1-pre` 的继续推进。`U1-pre` 解决的是“先搭统一骨架、先迁低风险文件”；本文件解决的是“在骨架基础上，真正把统一包结构与入口建立起来”。

---

## 2. 本次执行顺序

本次严格按以下顺序推进：

1. 先做 **S1**：审计 `tls_capture` 的运行时字段消费；
2. 再做 **S2**：统一 schema 与 `profile` 边界设计；
3. 再进入真正的 **U1**：整理包结构、`import`、`pyproject`、CLI；
4. **U2** 暂不落地逻辑，只建立 `VersionConfigLoader` 占位与后续路线；
5. 暂不收掉 `tls_capture/hooks/*.json` 的主运行路径。

这个顺序是为了避免：
- 先大改结构，再发现运行时需要的字段没搞清楚；
- 先删旧 JSON 路径，结果把运行时入口直接弄坏；
- 在 U1 阶段把 U2 的逻辑也硬塞进去，导致风险过高。

# 3. S1：`tls_capture` 运行时字段消费审计结果

本次重点阅读并核对了以下文件：

- `tls_capture/tls_capture.py`
- `tls_capture/lib/version_detect.py`
- `tls_capture/lib/correlator.py`
- `tls_capture/lib/output_writer.py`
- `tls_capture/hooks/chrome_hooks.js`
- `tls_capture/tools/fingerprint_scan.py`
- `tls_capture/tools/merge_analysis.py`

## 3.1 当前运行时链路

当前运行时主链路仍然是：

```text
tls_capture.py
  -> lib/version_detect.py
    -> 读取 hooks/*.json
    -> 把 JSON 注入 chrome_hooks.js
  -> Frida attach
  -> JS hook 发送 key
  -> correlator 做五元组关联
  -> output_writer 输出 keylog / wireshark log
```

因此当前最关键的事实是：

**`tls_capture` 当前运行时仍然强依赖本地 `hooks/*.json` 和 `chrome_hooks.js`。**

这也是为什么本阶段不能直接删掉旧 `hooks/` 路径。

---

## 3.2 `tls_capture` 实际消费的字段

### A. 必须字段：Frida 注入和运行时抓取直接依赖
从 `lib/version_detect.py::_build_hook_config()` 与 `chrome_hooks.js` 看，当前运行时直接消费：

#### 每个 hook 的必需字段
- `hook_points.prf.rva`
- `hook_points.key_expansion.rva`
- `hook_points.hkdf.rva`
- `hook_points.ssl_log_secret.rva`

#### 运行时模板必需字段
- `tls13_label_map`
- `tls13_key_len_offsets`
- `struct_offsets.ssl_st_rbio`
- `struct_offsets.bio_st_num`

这些字段在当前运行时是**必须**的，否则 Frida hook 无法正确工作。

---

### B. 运行时实际使用、但当前不一定都来自版本 JSON 的知识
从 `chrome_hooks.js` 看，当前还有一些逻辑直接写死在脚本中，例如：

- `readCR()` 路径
- `readCRSslLog()` 路径
- `master_secret` 校准逻辑
- 各种 label 行为判断

这些知识当前并没有全部从 JSON 注入，而是部分固化在 JS 中。

这说明：

**当前运行时真正消费的不是“完整的人工基线 JSON”，而是一部分 JSON 字段 + 一部分写在 JS 中的固定知识。**

这对 S2 的一个关键启发是：
- 不必把所有人工基线字段都塞回单版本 JSON；
- 应把跨版本稳定知识抽成 `profile` / 模板层。

---

### C. 非必须字段：更多是文档/追溯/验证用途
当前运行时并不直接依赖这些字段：

#### hook 语义与追溯字段
- `function_name`
- `ghidra_name`
- `fingerprint`
- `fingerprint_len`
- `role`
- `note`
- `source`
- `status`

#### 顶层验证字段
- `verified`
- `verified_method`
- `p3_capture_rate`
- `p4_tuple_hit_rate`
- `boringssl_commit`
- `ghidra_image_base`

这些字段对以下事情很重要：
- 数据库入库
- 审计追溯
- compare
- relocate
- 论文写作
- 人工 review

但**当前运行时 hook 本身并不直接消费它们**。

---

## 3.3 S1 结论

### 当前运行时的最小必要字段集合

#### 版本级最小必要字段
- `browser`
- `version`
- `platform`
- `arch`

#### hook 点最小必要字段
- `prf.rva`
- `key_expansion.rva`
- `hkdf.rva`
- `ssl_log_secret.rva`

#### 模板层最小必要字段
- `tls13_label_map`
- `tls13_key_len_offsets`
- `struct_offsets.ssl_st_rbio`
- `struct_offsets.bio_st_num`

### S1 的关键工程判断

1. 当前运行时并不需要完整分析 JSON 的所有字段；
2. 当前运行时也不应直接依赖“手工基线 JSON 的全部内容”；
3. `rva` 明显是**版本相关字段**，应来自数据库；
4. `tls13_label_map / struct_offsets / key_len_offsets` 更像**跨版本稳定模板字段**，应归到 `profile`；
5. `function_name / fingerprint / verified_method` 等应视为**追溯与验证层字段**，而不是运行时最低必需字段。

这就是后续 S2 的设计依据。

# 4. S2：统一 schema 与 profile 边界设计结果

## 4.1 三层分离架构

结合 S1 审计和现有代码现状，确定使用三层分离模型：

### 第一层：分析层（版本相关，入 DB）
由 `run.py / analyze.py / ingest.py` 负责，主要存：
- 版本元数据
- hook 点 RVA
- fingerprint
- 角色与追溯信息
- relocate 派生信息

### 第二层：运行时模板层（跨版本稳定，放 `profiles/`）
由未来 `profiles/*.json` 承载，主要存：
- `tls13_label_map`
- `tls13_key_len_offsets`
- `struct_offsets`
- `client_random` 路径
- 五元组策略说明
- 其他跨版本稳定的 TLS 栈知识

### 第三层：验证层（后处理写回）
由未来独立工具承载，主要存：
- `verified`
- `verified_method`
- `p3_capture_rate`
- `p4_tuple_hit_rate`
- 其他实验性验证结果

---

## 4.2 为什么不能把所有字段都继续塞进 `run.py`

因为当前已经很清楚：

- `run.py` 擅长的是“找 hook 点 + 抽指纹 + 输出基础 JSON”；
- 它不擅长也不应该承担：
  - 运行时模板知识
  - 五元组策略知识
  - 动态验证结果
  - 全量手工注释层

因此，统一 schema 必须保证：

### `run.py` 只负责分析层
最典型字段包括：
- `browser`
- `version`
- `platform`
- `arch`
- `tls_lib`
- `binary_sha256`
- `binary_size`
- `analysis_date`
- `analyzer_version`
- `ghidra_image_base`
- `hook_points.*.rva`
- `hook_points.*.fingerprint`
- `hook_points.*.fingerprint_len`
- `hook_points.*.ghidra_name`
- `hook_points.*.role`
- `hook_points.*.note`

### `profiles/` 负责模板层
最典型字段包括：
- `tls13_label_map`
- `tls13_key_len_offsets`
- `struct_offsets`
- `client_random`
- `five_tuple_strategy`

### `annotate_verified.py` 一类工具负责验证层
最典型字段包括：
- `verified`
- `verified_method`
- `p3_capture_rate`
- `p4_tuple_hit_rate`

---

## 4.3 当前阶段统一 schema 的实际设计结论

### hook_points 表继续以“版本相关、可入库”为核心
当前 DB 已较接近目标，已有字段包括：

- 未来只需补少量字段即可，不必推翻。

### versions 表当前可继续承载分析层元数据
已包含：
- `tls_lib_commit`
- `image_base`
- `binary_sha256`
- `binary_size`
- `analysis_date`
- `analyzer_version`
- `verified`

说明：
- 继续可用；

### `profile` 边界已明确
以下内容**不建议继续作为单版本 DB 字段主存储**，而更应进入 `profiles/`：
- `tls13_label_map`
- `tls13_key_len_offsets`
- `struct_offsets`
- `client_random`
- `five_tuple_strategy`

---

## 4.4 S2 阶段结论

S2 的核心不是把所有字段都落代码，而是把边界立住：

### 分析层（DB）
存版本相关、变动频繁、可自动产出的字段。

### profile 层（文件模板）
存跨版本稳定、运行时强依赖、但不适合每个版本重复保存的字段。

### 验证层（后处理）
存实验结果、校验结果和可信度信息。

这个边界已经足够支撑进入真正的 U1。
