## 1. 给 Claude Code 的简短项目介绍

你正在接手一个长期工程项目，而不是一次性脚本任务。

项目的核心目标是：

**建立一个面向浏览器版本的 TLS 密钥指纹数据库系统。**

这个系统围绕以下能力构建：
- 静态分析浏览器二进制；
- 识别关键 TLS hook 点；
- 记录函数 fingerprint / 特征码 / RVA；
- 将这些信息和浏览器版本绑定后存入数据库；
- 运行时按版本读取这些数据进行 hook；
- 抽取 TLS 密钥并完成五元组关联；
- 通过 relocate 适配小版本偏移；
- 若 relocate 不足，则退回逆向分析；
- 持续扩充并维护数据库。

当前主要分成两个仓库：
- `TSHunter`：分析、relocate、入库、知识生产
- `p_t_c`：运行时查询、消费数据库、执行 hook、输出关联结果

你的任务不是只修一个 bug，而是帮助把这套系统逐步做完整。

---

## 2. 项目当前判断

### 2.1 `TSHunter` 当前状态
已具备：
- Chrome/BoringSSL 主链路的核心 hook 点自动识别；
- SQLite 数据库基础模型；
- `ingest/query/capture/relocate` 最小闭环；
- `PARTIAL/OK/FAIL` relocate 第一版；
- DB miss 时优先尝试 relocate；
- relocate 不足时回退完整分析。

当前不足：
- 自动分析 JSON 与手工基线 schema 不统一；
- hook 点结果缺少语义、参数、运行时知识层；
- 验证层数据未形成独立工程流程；
- 与 `p_t_c` 的联动尚未真正闭环。

### 2.2 `p_t_c` 当前状态
用户判断它的作用是：
- 识别目标浏览器；
- 从数据库中获取对应 hook 数据；
- 若无数据，则触发分析链路；
- 若有数据，则直接读取并运行时 hook；
- 按用户模式执行；
- 支持批处理模式；
- 最终服务于密钥提取与五元组关联。

当前优先任务不是立刻大改它，而是先确认：
- 它当前如何识别浏览器；
- 它当前需要什么 hook 数据结构；
- 它是否已经有数据库查询接口；
- 它的运行时配置格式是什么；
- 它和 `TSHunter` 的接口边界在哪里。

---

## 3. 总体策略：先消费端，后生产端，再整合

用户当前提出的顺序是：
1. 检查升级 `p_t_c`
2. 得到这次长跑结果后检查升级 `TSHunter`
3. 两项目放入一个文件夹整合
4. 在服务器批量下载/分析 Chrome 各版本
5. 用首批数据扩展到其他浏览器

这是合理的，但还可以更工程化地细化。

### 更完整的推荐顺序
1. 先梳理 `p_t_c` 的真实消费需求；
2. 用消费需求反推 `TSHunter` 的统一 schema；
3. 修正 `TSHunter` 的输出、入库、查询；
4. 让两仓在单浏览器单版本上形成闭环；
5. 再做统一工作目录整合；
6. 再上服务器做 Chrome 批量分析；
7. 再扩展到其他浏览器与 TLS 库。

这条路线可以最大限度降低返工。

---

## 4. 最高优先级目标

### 目标 1：定义统一数据契约
这是当前第一优先级。

必须先明确：
- `TSHunter` 输出什么 JSON；
- 数据库如何表示 hook 点；
- `p_t_c` 消费什么字段；
- 运行时 hook 需要哪些参数；
- 验证信息如何表达；
- relocate 派生结果如何表达。

如果没有统一契约，后续批量分析出来的数据也很难稳定消费。

---

## 5. 面向 Claude Code 的工作原则

### 原则 1：不要一上来就重构所有代码
优先做：
- 理清边界
- 定义 schema
- 打通单条主链路

### 原则 2：优先保证 Chrome/BoringSSL/Linux/x86_64 主链路
这条主链路必须先稳定到：
- 可分析
- 可入库
- 可查询
- 可 hook
- 可验证
- 可五元组关联

### 原则 3：分析层与验证层分离
- 静态分析结果 ≠ 已验证结果
- relocate 命中 ≠ 实际抓取可靠
- 必须保留独立验证层

### 原则 4：以 `p_t_c` 的真实消费为导向
不要让 `TSHunter` 闭门设计字段。先看运行时到底需要什么，再反推输出结构。

### 原则 5：先形成系统，再追求扩展
先把 Chrome 一条线做成产品级闭环，再考虑多浏览器扩展。

## 6. 分阶段执行计划

# Phase 1：审查并升级 `p_t_c`

## 目标
明确运行时消费端的真实需求，为统一 schema 提供依据。

## Claude Code 应做的事
1. 通读 `p_t_c` 仓库关键入口；
2. 识别浏览器检测逻辑；
3. 识别数据库读取逻辑；
4. 识别 hook 配置格式；
5. 识别五元组关联逻辑；
6. 列出它真正需要的字段清单；
7. 输出一份“消费端接口需求说明”。

## 本阶段产物
- `p_t_c` 现状分析文档
- 运行时最小字段需求列表
- `TSHunter` / `p_t_c` 对接缺口清单

## 关键问题
- `p_t_c` 当前是不是已经支持数据库驱动？
- 它是按浏览器名、版本、hash 还是路径识别目标？
- 它期望拿到的是 RVA，还是完整 hook 模板？
- 它如何处理 TLS 1.2 / 1.3 的不同 key 提取逻辑？
- 五元组关联是已有实现还是待设计？

---

# Phase 2：根据消费需求升级 `TSHunter`

## 目标
让 `TSHunter` 成为真正稳定的“知识生产端”。

## Claude Code 应做的事
1. 对比自动分析 JSON 与手工基线 JSON；
2. 定义统一 schema；
3. 修改 `run.py` 输出结构；
4. 修改 `tools/ingest.py` / `tools/query.py` 兼容统一结构；
5. 明确 `verified` 与验证层表达方式；
6. 决定哪些运行时知识放进结果，哪些放入模板层。

## 本阶段建议的关键输出
- 统一 JSON schema 文档
- 数据库字段映射说明
- `run.py` 输出升级
- `ingest/query` 兼容性修正
- 验证注释后处理脚本设计

## 当前明确已知的改进方向
- `function` 拆分为 `function_name` + `ghidra_name`
- 统一 `meta` 字段命名
- 规范 `role` / `note`
- 为 hook 结果预留 `params` / `read_on` / `output_len`
- 为顶层预留 `verified` / `verified_method`

---

# Phase 3：在单版本上打通两仓闭环

## 目标
让一个已知版本 Chrome 可以完成从识别到运行时 hook 的闭环。

## 理想闭环
1. `p_t_c` 识别目标浏览器；
2. 查询数据库；
3. 命中对应版本 hook 数据；
4. 生成运行时配置；
5. 执行 hook；
6. 提取 TLS 密钥；
7. 完成五元组关联；
8. 校验结果有效性。

## Claude Code 应做的事
1. 选择 `Chrome 143.0.7499.169` 作为首个标准样本；
2. 确保 `TSHunter` 输出能被 `p_t_c` 正常读取；
3. 在 `p_t_c` 内增加或修正数据库读取入口；
4. 如果需要，补一个轻量中间适配层；
5. 完成一次标准闭环验证。

## 本阶段产物
- 一条真正跑通的 Demo 路径
- 一份闭环验证文档
- 一组可复现命令

---

## Claude Code 应做的事

2. 决定共享数据库位置；
3. 决定共享 schema 文档位置；
4. 决定哪些脚本共享、哪些各自保留；
5. 写整合说明文档。

## 注意事项
- 不要过早把两个仓库硬合并成一个大仓；
- 先统一工作目录和接口，再考虑更深层整合。

# Phase 5：服务器侧 Chrome 批量分析

## 目标

## Claude Code 应做的事

2. 设计服务器批量任务脚本；
3. 设计结果存放结构；
4. 设计失败重试与日志收集机制；
5. 设计批量入库与去重规则；

## 注意事项

# Phase 6：五元组关联与运行时取证能力完善

## 目标
把项目从“能找到 hook”推进到“能形成可用取证结果”。

## Claude Code 应做的事
1. 明确五元组数据来源；
2. 明确连接标识与 TLS key 的关联逻辑；
3. 定义记录结构与存储方式；
4. 明确可信度模型；
5. 输出标准化关联结果。

## 本阶段关键问题
- 五元组来自 eBPF、socket 追踪、时间相关性还是其他方案？
- client_random 与连接标识如何稳定绑定？
- 离线取证时如何重建证据链？
