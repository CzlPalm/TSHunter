## 3. 当前判断

## 3.1 关于小样本：“没有 `[RESULT]`” 目前不能直接判定为脚本 bug

这点非常关键。

从现有证据看，第二次 smoke test 的日志本身说明：

- 它不是一个已经进入 `ghidra_analysis.sh` 正常执行流程的有效样本运行

- 因此这次“没有 `[RESULT]`”的结论并不成立

所以当前不能下结论说：

- `TLShunterAnalyzer.java` 逻辑错误
- `ghidra_analysis.sh` 参数错误
- `run.py` 解析逻辑仍有 bug

**更准确的结论是：当前 smoke test 结果无效，需要重新做一次干净、可验证的小样本测试。**

## 3.2 关于 Chrome 长跑：当前状态是“正在正常推进”，不是“已经卡死”

根据 monitor 信息，当前 Chrome 长跑应继续保留，不建议中止。理由：

因此目前没有证据表明：

当前最合理的策略是：

## 4. 对第一次“几秒退出”根因的进一步收敛

截至目前，已经能确认两类独立问题：

### 根因 A：镜像标签不存在
第一次 smoke test 已直接证明：

- `docker run tlshunter-integrated:phase2`
- 但本地没有该 tag
- Docker 尝试远程拉取失败
- 因而秒退

这可以解释“秒退”的一种根因。

### 根因 B：镜像状态不干净 / 使用了中间态镜像
第二次 smoke test 的日志说明：

- 启动后看到的是 Ghidra 下载、解压过程
- 而不是最终成品镜像执行 `ghidra_analysis.sh`

这意味着之前为了补救 tag 缺失而采用的“从运行中容器 commit 成镜像”的方式，带来了副作用：

- 该镜像很可能只是构建过程中的中间状态
- 并不等于 Dockerfile 完整构建完成后的最终镜像
- 用它做 smoke test，结论不可信

也就是说：

> 目前已经可以把“没有 `[RESULT]`”与“脚本识别失败”区分开：当前更大的问题其实是镜像生命周期和镜像一致性，而不是识别逻辑本身已经被证伪。

## 9. 2026-04-17 修复记录：Ghidra 内存参数与日志可见性

### 9.1 堆大小问题确认与修复

已确认 `analyzeHeadless` 仅传入 `-maxMem 16G` 不能可靠保证最终 JVM 以 16G 堆启动。
原因是 `analyzeHeadless -> launch.sh -> java` 链路中，`launch.sh` 可能仍把默认值带成 `-Xmx2G`。

为避免 Chrome 大样本在 import 阶段因 2G 堆导致极重 GC、长期停留在导入阶段，已修改 `integrated/ghidra_analysis.sh`，在调用前增加双保险：

- `export MAXMEM=16G`
- `export _JAVA_OPTIONS="-Xmx16G"`

同时，将 headless 脚本参数改为按文件名传递，而不是绝对路径：

- `-prescript MinimalAnalysisOption.java`
- `-postScript TLShunterAnalyzer.java`

当前脚本关键调用如下：

```24:31:integrated/ghidra_analysis.sh
export MAXMEM=16G
export _JAVA_OPTIONS="-Xmx16G"
/opt/ghidra_12.0.3_PUBLIC/support/analyzeHeadless /tmp "tlshunter_$(date +%s)" \
    -import "$selected_bin" \
    -scriptPath /usr/local/src \
    -prescript MinimalAnalysisOption.java \
    -postScript TLShunterAnalyzer.java \
    -maxMem 16G | tee /host_output/analysis.log
```

### 9.2 为什么之前“看不到任何输出”

后续最小化验证证明，问题并不在 `tee` 或 shell 重定向，而在日志配置：

- 容器内 `custom_log4j.xml` 的 Root level 原本是 `ERROR`
- 这会压掉 Ghidra headless 的正常分析日志、脚本执行日志和大部分可观测输出
- 表面现象就变成：只有 JVM 版本信息，几乎看不到导入、分析、脚本执行过程

已将日志级别修复为 `INFO`：

```8:10:/home/palm/TLSHunter/TLSKeyHunter/custom_log4j.xml
<Loggers>
    <Root level="INFO">
        <AppenderRef ref="Console"/>
```

这一步是本次 smoke 成功恢复可观测性的关键修复。

## 11. Chrome 重跑时应重点关注的日志信号清单

建议重跑 Chrome 时，不要只盯最终是否有 `[RESULT]`，而是按阶段检查以下信号。

### 11.1 启动阶段：确认镜像和 JVM 参数正确

应该看到：

- `[*] Analyzing chrome`
- `Picked up _JAVA_OPTIONS: -Xmx16G`
- `Using log config file: file:/usr/local/src/custom_log4j.xml`
- `Headless startup complete`

如果缺少这些信号，优先怀疑：

- 运行的不是最新镜像
- 容器没有带上最新 `ghidra_analysis.sh`
- JVM 堆参数没有成功注入

### 11.2 导入阶段：确认真的进入 Ghidra 导入流程

应该看到：

- `HEADLESS: execution starts`
- `Creating project: /tmp/...`
- `IMPORTING: file:///usr/local/src/binaries/...`
- `Using Loader: Executable and Linking Format (ELF)` 或对应格式
- `Loading file:///...`

如果只有 JVM 版本信息，没有这些行，说明仍然没有真正进入 headless 主流程。

### 11.3 预分析脚本阶段：确认 `MinimalAnalysisOption.java` 已执行

应该看到：

- `REPORT: Execute script: MinimalAnalysisOption.java`
- `SCRIPT: /usr/local/src/MinimalAnalysisOption.java`

如果缺少这两行，说明 prescript 没有执行成功。

### 11.4 自动分析阶段：确认 Ghidra 仍在推进，不是“假活着”

大型 Chrome 分析时，预期会看到：

- `ANALYZING all memory and code: ...`
- 各类分析器计时信息
- 大量导入 / 修复 / 噪声日志，例如：
  - `Invalid PNG data ...`
  - `Invalid GIF data ...`
  - `function body repair failed due to overlap ...`

这些噪声本身**不等于失败**。真正需要关注的是：

- 日志是否仍持续增长
- CPU / I/O 是否仍持续变化
- 是否最终出现分析总耗时汇总

### 11.5 主分析脚本阶段：确认 `TLShunterAnalyzer.java` 已真正执行

应该看到：

- `REPORT: Execute script: TLShunterAnalyzer.java ...`
- `SCRIPT: /usr/local/src/TLShunterAnalyzer.java`
- `TLShunterAnalyzer.java> TLShunter integrated analyzer v...`
- `TLShunterAnalyzer.java> [*] Binary: chrome`

如果到了导入/分析完成后仍没有这些行，说明 postScript 没有真正执行。

### 11.6 结果阶段：最终成功判据

至少应看到以下之一：

- `TLShunterAnalyzer.java> [RESULT] type=HKDF ...`
- `TLShunterAnalyzer.java> [RESULT] type=PRF ...`
- `TLShunterAnalyzer.java> [RESULT] type=SSL_LOG_SECRET ...`
- `TLShunterAnalyzer.java> [RESULT] type=KEY_EXPANSION ...`

最后应看到：

- `REPORT: Post-analysis succeeded for file: ...`
- `REPORT: Save succeeded for: ...`
- `REPORT: Import succeeded`

### 11.7 需要立刻警惕的失败信号

若出现以下情况，应优先判为异常并立即复查：

1. 只有 JVM 版本信息，没有任何 `HEADLESS:` / `IMPORTING:` 日志
2. 没有 `SCRIPT:` 行
3. 容器很快退出，且没有 `[RESULT]`
4. 容器退出码非 `0`
5. 出现明确 Java 异常栈，如：
   - `Exception`
   - `Caused by:`
   - `OutOfMemoryError`
6. 长时间没有日志增量，同时 CPU / I/O 也接近静止
